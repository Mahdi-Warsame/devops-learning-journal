### Problem

```text

```

while running AWS commands.

### Cause

The configured AWS credentials were invalid, expired, or belonged to a different AWS account.

### Troubleshooting

Checked configuration:

```bash
aws configure list

```

Verified:

- Access Key
- Secret Key
- AWS Region
- Active profile

### Solution

Updated AWS credentials using:

```bash
aws configure
```

or replaced the credentials file with valid IAM user credentials.

## 3. Docker Login to Amazon ECR

### Challenge

Docker cannot directly log into Amazon ECR using username/password.

### Solution

Authenticate Docker using AWS CLI:

```bash
aws ecr get-login-password \
| docker login \
--username AWS \
--password-stdin <account-id>.dkr.ecr.<region>.amazonaws.com
```

This generates a temporary authentication token for Docker.

# Docker Compose

## What is Docker Compose?

Docker Compose is a tool that allows you to define and run multiple Docker containers using a single configuration file (`docker-compose.yml`).

Instead of starting each container manually, Docker Compose automates the entire process.

For my Flask and MySQL application, Docker Compose automatically:

* Created a dedicated Docker network.
* Started both the Flask application container and the MySQL database container.
* Connected both containers to the same network.
* Allowed the Flask application to communicate with MySQL using the service name.
* Managed all services with a single command.

```bash
docker compose up
```

---

## `build:` vs `image:`

One important concept I learned is the difference between `build` and `image`.

### Build an image

```yaml
services:
  web:
    build: .
```

Docker looks for a `Dockerfile` in the current directory and builds a new image before starting the container.

---

### Use an existing image

```yaml
services:
  web:
    image: my-flask-app:multistaged
```

Docker skips the build process and simply starts a container using the existing image.

---

## Docker Compose Networking

I learned that:

* Images do **not** communicate with each other.
* **Containers** communicate with each other.
* Docker Compose automatically creates a private network for all services.
* Containers can communicate using their service names.

For example:

```yaml
services:
  web:
    ...

  mydb:
    ...
```

The Flask application can connect to MySQL using:

```text
mydb
```

instead of using an IP address.

# Multi-Stage Builds

## Why Multi-Stage Builds?

Initially, my Docker image contained everything needed to build the application, including:

* gcc
* python3-dev
* libmariadb-dev
* pkg-config

These packages are necessary during the build process but unnecessary once the application is running.

Keeping them in the final image increases:

* Image size
* Build time
* Security risks (more software means a larger attack surface)

Multi-stage builds solve this problem by separating the **build environment** from the **runtime environment**.

---

# Understanding Build Stages

## Build Stage

```dockerfile
FROM python:3.8-slim AS build
```

This stage is responsible for:

* Installing development tools
* Installing Python packages
* Compiling dependencies
* Preparing the application

---

## Runtime Stage

```dockerfile
FROM python:3.8-slim
```

This starts a completely new image.

This was the biggest lesson I learned.

Every `FROM` instruction creates a **brand-new image**.

Nothing from the previous stage exists unless it is explicitly copied.

---

# Mistake I Made

Initially, my runtime stage only copied the application source code.

```dockerfile
COPY --from=build /app /app
```

This copied:

* `app.py`

But it did **not** copy:

* Flask
* mysqlclient
* Other installed Python packages

The application failed with:

```text
ModuleNotFoundError: No module named 'flask'
```

---

# First Fix

I copied the installed Python packages.

```dockerfile
COPY --from=build /usr/local /usr/local
```

This copied everything installed by `pip`, including:

* Flask
* mysqlclient
* Python executables

This resolved the Flask import error.

---

# Second Problem

After fixing Flask, I encountered another error:

```text
ImportError: libmariadb.so.3: cannot open shared object file
```

I learned that Python packages are not always self-contained.

The `mysqlclient` package depends on an operating system shared library (`libmariadb.so.3`).

Although I copied the Python package, the runtime image still lacked the required system library.

Installing the runtime package in the final stage solved the issue.

---

# My Mental Model

One of the most valuable lessons I learned is how Docker builds images.

A Docker image is a snapshot of a filesystem.

Docker processes the Dockerfile **one instruction at a time**, creating a new image layer for each instruction.

In a multi-stage build:

1. Docker builds the first image (Build Stage).
2. Docker starts a completely new image (Runtime Stage).
3. Only the files explicitly copied using `COPY --from=build` are transferred into the final image.

Everything else from the build stage is discarded.

---

# Key Takeaways

* A Docker image is a snapshot of a filesystem.
* Docker builds images layer by layer.
* Every Dockerfile instruction creates a new image layer.
* `RUN` executes commands during the image build process.
* `COPY` copies files into the image.
* `docker build` creates an image.
* `docker run` creates and starts a container from an image.
* Every `FROM` instruction starts a completely new image.
* Multi-stage builds separate build dependencies from runtime dependencies.
* Build dependencies are only needed while building the application.
* Runtime dependencies are required when the application is running.
* Docker Compose simplifies running multiple containers together.
* Docker Compose automatically creates networks for services.
* Containers communicate using service names defined in `docker-compose.yml`.
